/**
 * Schedule weekly metagame refresh with node-cron.
 *
 * Per CONTEXT.md D-02: Weekly Sunday refresh, dedicated cron job
 * (NOT interleaved inside executePriceCollection).
 *
 * Default schedule: '0 2 * * 0' = 2:00 AM every Sunday.
 * Override via process.env.CRON_METAGAME_REFRESH.
 *
 * The cron callback invokes executeMetagameRefresh() from @/scraper/metagame
 * and logs the returned summary. Defensive try/catch ensures the scheduler
 * never crashes even if the orchestrator throws.
 *
 * @returns Object with start() and stop() methods, mirroring schedulePriceCollection
 *
 * @example
 * ```ts
 * const refresher = scheduleMetagameRefresh()
 * refresher.start() // begin weekly cron
 * refresher.stop()  // halt weekly cron (e.g., on bot shutdown)
 * ```
 */
export function scheduleMetagameRefresh(): {
  start: () => void
  stop: () => void
} {
  const schedule = process.env.CRON_METAGAME_REFRESH || '0 2 * * 0' // Sunday 2:00 AM

  logger.info(`Scheduling metagame refresh: schedule=${schedule}`)

  metagameJob = cron.schedule(
    schedule,
    async () => {
      logger.info('Weekly metagame refresh triggered')
      try {
        const summary = await executeMetagameRefresh()
        logger.info(`Weekly metagame refresh complete: ${JSON.stringify(summary)}`)
      } catch (error) {
        const message = error instanceof Error ? error.message : String(error)
        logger.error(`Weekly metagame refresh threw an error (scheduler continues): ${message}`)
      }
    },
    { scheduled: false },
  )

  return {
    start: () => {
      logger.info('Starting metagame refresh scheduler')
      metagameJob?.start()
    },
    stop: () => {
      logger.info('Stopping metagame refresh scheduler')
      metagameJob?.stop()
    },
  }
}
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| process.env → cron schedule string | Untrusted env input could contain a malformed cron expression — node-cron will throw at registration. |
| Weekly cron callback → executeMetagameRefresh | Inherited from Plan 04: orchestrator's internal try/catch returns a summary on errors. This plan adds an OUTER try/catch as defense in depth. |
| Scheduler exports → bot/server bootstrap | Public API surface; misuse (e.g., calling start() twice) does not pose security risk in v1. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-05-01 | Denial of Service | Malformed CRON_METAGAME_REFRESH env var crashes scheduler at boot | mitigate | node-cron's `cron.schedule` validates the expression and throws on registration. The bot bootstrap can wrap scheduleMetagameRefresh() in try/catch (caller responsibility). Default `'0 2 * * 0'` is hardcoded fallback when env var is unset. |
| T-5-05-02 | Tampering (data destruction) | Cron firing more often than weekly destroys/recreates wishlist rows excessively | mitigate | Default schedule is weekly (`0 2 * * 0`). Even if misconfigured to "every minute", the orchestrator's removal logic STILL filters on `isAutoAdded=true` (Plan 04) so user-added rows are never destroyed. Operational risk only — not a security risk. |
| T-5-05-03 | Information Disclosure | Logged summary contains card names | accept | logger.info logs a JSON summary with addedCount, removedCount, perFormat counts (numbers only) — NOT card names. The schedule string is logged in plaintext but is not sensitive. |
| T-5-05-04 | Tampering | Multiple invocations of scheduleMetagameRefresh leak orphaned ScheduledTask objects | accept | Existing scheduler pattern (schedulePriceCollection lines 168-170) does NOT guard against double-registration; this is consistent house style. Bot bootstrap calls each scheduler exactly once. Operational concern only. |
| T-5-05-05 | Repudiation | No log when cron is registered or fires | mitigate | Three log points: registration (`Scheduling metagame refresh: schedule=...`), trigger (`Weekly metagame refresh triggered`), completion (`Weekly metagame refresh complete: <summary>`). |
| T-5-05-06 | Spoofing | Env var injection from .env.local with malicious schedule | accept | Single-user mode; .env.local is not user-supplied data. .env.example documents the safe default. |
| T-5-05-07 | Denial of Service | Scheduler callback throws unhandled rejection | mitigate | Outer try/catch in the cron callback (in addition to orchestrator's inner try/catch). Verified by test "cron callback does NOT throw". |
| T-5-05-08 | Cron drift / time confusion | TZ env var unset causes cron to fire at unexpected UTC time | accept | node-cron uses the system TZ. Default `'0 2 * * 0'` is interpreted in server local time. Operational concern; documented in code comment ("Sunday 2:00 AM"). v2 may add explicit TZ handling. |
