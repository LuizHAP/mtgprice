**Part B — Create `src/db/schema/detectionCandidates.ts`** — the state store required by D-07's two-consecutive-runs semantics. A candidate row exists for a `(card_id, source)` pair IFF the previous detection run fired for that pair but the opportunity was NOT yet promoted (because it was the first confirming run). On the second consecutive confirming run, the detector promotes the candidate to an `opportunities` row and deletes the candidate. If a run does NOT confirm the drop, the candidate row is deleted (streak broken).

Schema (use the exact shape below):

```typescript
import { pgTable, serial, varchar, timestamp, unique, index } from 'drizzle-orm/pg-core'
import { cards } from './cards'

export const detectionCandidates = pgTable(
  'detection_candidates',
  {
    id: serial('id').primaryKey(),
    cardId: varchar('card_id', { length: 255 })
      .notNull()
      .references(() => cards.oracleId, { onDelete: 'cascade' }),
    source: varchar('source', { length: 20 }).notNull(),
    firstSeenAt: timestamp('first_seen_at').notNull().defaultNow(),
  },
  (table) => ({
    uniqueCardSource: unique('detection_candidates_card_source_key').on(table.cardId, table.source),
    firstSeenIdx: index('detection_candidates_first_seen_idx').on(table.firstSeenAt),
  }),
)
```

Notes:
- UNIQUE `(card_id, source)` — there is at most ONE candidate row per pair. Plan 03's queries rely on this for idempotent `insertCandidate`.
- `ON DELETE CASCADE` on the FK — if a card is removed, dangling candidates are cleaned up automatically.
- `first_seen_at` index — used by Plan 03's `deleteStaleCandidates()` housekeeping query (delete rows older than 2 days).
- No relations export needed for this table; downstream code treats it as an opaque state table.
