<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `process.env` → config module | Env vars are operator-controlled and trusted, but CAN be malformed (typos, unset). |
| config module → downstream callers | Pure TypeScript object returned; no I/O. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | Input validation | `parseFloat('abc')` → NaN propagates | mitigate | `Number.isFinite(parsed)` check before accepting; on failure, fallback to default + `logger.warn`. Verified by Test 5 and Test 9. |
| T-04-02-02 | Input validation | Negative / zero day counts | mitigate | Post-parse guard `parsed >= 1` (or `> 0` for threshold); fallback + warn. Verified by Test 6 and Test 7. |
| T-04-02-03 | Denial of Service | Thresholds set to absurd values (e.g., DROP_THRESHOLD=0.001 causing every price to fire) | accept | Operator-controlled env vars; no DoS surface. Out-of-spec tuning is the operator's responsibility, and the 7-day cooldown from D-12 caps the blast radius. |
| T-04-02-04 | Information Disclosure | Logger prints raw env var value | mitigate | Warning messages log the env var NAME and raw value, which is not a secret (0.15, 7, etc.). Do NOT log DATABASE_URL, TELEGRAM_BOT_TOKEN, JWT_SECRET — none of those are read by this module. Verified by grep: `process.env.DETECT_` and `process.env.CRON_` only. |
| T-04-02-05 | Tampering | Mutating the returned config object affects other callers | mitigate | `loadDetectionConfig` returns a fresh object on every call (re-reads env). Callers that mutate get their own copy. No shared singleton state. |
| T-04-02-06 | SQL injection | N/A | N/A | This module has zero DB access. |
| T-04-02-07 | Telegram message injection | N/A | N/A | This module has zero Telegram I/O. Plan 05 pipes the config object into a static template string — escaping is Plan 05's concern. |
| T-04-02-08 | Authorization | Module readable by bot command handler | accept | The bot command handler runs behind Phase 1's chat-ID whitelist; `/config` read-only output is safe to emit to the whitelisted user. |

## Security Constraints Coverage

1. **Input handling** — Core purpose of this plan: validate DETECT_* numerics with NaN/negative guards.
2. **Authorization** — No new auth; downstream /config stays behind whitelist.
3. **Secrets** — Module only reads `DETECT_*` and `CRON_*` vars. Does NOT touch TELEGRAM_BOT_TOKEN, JWT_SECRET, DATABASE_URL.
4. **SQL injection** — N/A (no DB).
5. **Telegram message injection** — N/A (no Telegram).
6. **Error disclosure** — Warnings go through Winston, not to Telegram.
7. **Denial-of-service** — Per T-04-02-03.
</threat_model>
