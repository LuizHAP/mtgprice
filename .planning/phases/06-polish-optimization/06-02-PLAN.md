<objective>
Refactor the price-collection orchestrator so it (a) runs cards through a `p-limit`-controlled queue instead of the current sequential `for` loop, and (b) wraps every per-source raw fetch with `withRetry` (Plan 01) BEFORE the circuit breaker — implementing D-05/D-06/D-07/D-08/D-09. This is what unlocks the >95% success rate AND the 1000+ card target without exceeding rate limits.

Purpose: With 1000 cards × 4 sources = 4000 calls executed sequentially at ~1 req/sec, a single collection run takes ~66 minutes. Per-source concurrency of 5 brings worst-case to ~13 minutes while staying inside LIGAMAGIC (30/min) and TCGPLAYER (40/min) presets. Retry wrapping makes the success rate climb past 95% by absorbing transient 429/timeout/network errors before they count against the circuit breaker.

Output: Modified `src/scraper/orchestrator.ts` with p-limit + retry composition, new active tests in `src/scraper/__tests__/orchestrator.test.ts`, p-limit added to package.json, and three new env-var defaults documented in .env.example.

    ```typescript
    export async function fetchAllPrices(oracleIds: string[]): Promise<FetchAllPricesStats> {
      const stats: FetchAllPricesStats = {
        total: oracleIds.length,
        fetched: 0,
        skipped: 0,
        failed: 0,
        errors: [],
      }

      logger.info(`Starting price collection for ${oracleIds.length} cards (concurrency=${CONCURRENCY_PER_SOURCE})`)

      const limit = pLimit(CONCURRENCY_PER_SOURCE)

      const tasks = oracleIds.map((oracleId, i) =>
        limit(async () => {
          try {
            const results = await fetchCardPriceFromAllSources(oracleId)
            const successCount = Object.values(results).filter((r) => r.success).length
            const skipCount = Object.values(results).filter((r) => r.error?.includes('Skipped')).length
            stats.fetched += successCount
            stats.skipped += skipCount
            if ((i + 1) % 10 === 0) {
              logger.info(`Progress: ${i + 1}/${oracleIds.length} cards processed`)
            }
          } catch (error) {
            stats.failed += 1
            const errorMsg = error instanceof Error ? error.message : String(error)
            stats.errors.push(`${oracleId}: ${errorMsg}`)
            logger.error(`Failed to fetch prices for ${oracleId}: ${errorMsg}`)
          }
        }),
      )

      await Promise.all(tasks)

      logger.info(
        `Price collection complete: ${stats.fetched} fetched, ${stats.skipped} skipped, ${stats.failed} failed`,
      )
      if (stats.errors.length > 0) {
        logger.warn(`Encountered ${stats.errors.length} errors during collection`)
      }
      return stats
    }
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server → external scraping/API endpoints | Each source fetcher already crosses this boundary (existing). This plan changes the timing and concurrency of crossings, not the boundary itself. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-02-01 | Denial of Service | Concurrent fetches saturate own outbound bandwidth or trip per-source rate limits | mitigate | Cap CONCURRENCY_PER_SOURCE at 5 (env tunable). Each source already runs through `checkRateLimitPreset` (rate-limiter.ts) which enforces token bucket. 5 concurrent × 4 sources = max 20 in-flight, well under combined preset budget. |
| T-06-02-02 | Denial of Service | Retry layer amplifies failed-source load (3 attempts × 1000 cards × 4 sources) | mitigate | withRetry uses exponential backoff (1s/2s) which spreads attempts in time. The circuit breaker layer (out of scope here) opens after 50% failure rate over the 10s rolling window, halting further retries on truly down sources. |
| T-06-02-03 | Tampering | SCRAPER_CONCURRENCY_PER_SOURCE / SCRAPER_RETRY_* read from process.env | accept | Env vars are server-controlled. Defaults `|| 5`, `|| 3`, `|| 1000` neutralise NaN/empty values. No security boundary impact. |
| T-06-02-04 | Information Disclosure | Error messages from external sources stored in stats.errors | accept | Errors are written to Winston logs and the in-memory stats array (returned to scheduler, not exposed via HTTP). No PII in scraper errors. |
| T-06-02-05 | Repudiation | Concurrent writes via insertPrice could race | accept | TimescaleDB hypertable on (card_id, timestamp) tolerates same-millisecond inserts; existing schema already supports concurrent writes (Phase 2 was tested under parallel international fetches). p-limit only reorders timing, not semantics. |
</threat_model>
